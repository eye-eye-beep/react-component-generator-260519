변경사항을 분석하여 컨벤션에 맞게 커밋합니다.

## 절차

### 1. 변경사항 분석
다음 명령으로 현재 상태를 파악합니다:
- `git status` — 스테이징 상태 확인
- `git diff` — 워킹트리 변경사항
- `git diff --staged` — 이미 스테이지된 변경사항
- `git log --oneline -5` — 최근 커밋 스타일 참고

### 2. 논리적 단위 분류
변경사항을 아래 기준으로 묶습니다:
- **기능 추가** → `feat`
- **버그 수정** → `fix`
- **코드 정리/구조 개선** → `refactor`
- **스타일/포맷** → `style`
- **설정/의존성/문서** → `chore`

여러 성격의 변경이 섞여 있으면 논리적으로 분리 가능한지 판단하고, 가능하면 복수 커밋을 제안합니다.

### 3. 한국어 커밋 메시지 작성
형식: `타입: 무엇을 왜 했는지 간결하게`

예시:
```
feat: 다크 테마 CSS 변수 및 색상 팔레트 추가
fix: 모바일에서 버튼 레이아웃 깨지는 문제 수정
refactor: App.css 중복 색상값 CSS 변수로 통일
chore: Tailwind CSS v4 의존성 추가
```

규칙:
- 제목은 50자 이내, 명령형으로
- 변경 이유가 명확하면 본문에 추가
- `Co-Authored-By` 푸터 포함

### 4. 사용자 승인 후 커밋
제안한 커밋 메시지와 포함될 파일 목록을 보여주고 승인을 요청합니다.
승인 후에만 `git add` 및 `git commit`을 실행합니다.

> 파일이 `.env`, 시크릿, 바이너리를 포함하면 제외하고 경고를 표시합니다.
